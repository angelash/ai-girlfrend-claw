# AI-Girlfriend 对接需求分析

## 1. 背景与目标

当前 UE 客户端（Lobster 模式）需要一个稳定的本地适配层，将游戏内协议转为 OpenClaw 会话路由能力，满足：

- 聊天链路稳定可用（可观测、可排障）
- 人设与上下文可控（优先使用 OpenClaw 会话体系）
- UE 侧接入成本最低（尽量不改客户端协议）

## 2. 业务目标

1. **保持 UE 协议不变**
   - 客户端继续使用 `43113` 的 HTTP/WS 接口。
2. **统一能力入口**
   - `ai-girlfriend` 作为主维护模块，承担核心业务逻辑。
3. **会话可追踪**
   - 具备可定位的会话 ID；支持固定单会话，便于历史查询。
4. **结构化回包**
   - 返回 `text/emotion/actions/subtitleTimeline`。

## 3. 非目标（当前阶段）

- 不在本阶段实现完整 TTS/音频合成闭环。
- 不在本阶段重构 UE 客户端协议。
- 不追求和所有 OpenClaw 会话存储入口“完全同构”（先保证可查可用）。

## 4. 角色与使用场景

### 4.1 角色
- **UE 客户端**：发起登录、进场、聊天、语音相关协议。
- **适配层（ai-girlfriend + 内嵌 adapter）**：协议转换、会话路由、结构化回包。
- **OpenClaw 会话层**：人设、上下文、模型调用。

### 4.2 核心场景
1. 玩家进入龙虾模式（continue/new game）。
2. 玩家发送聊天消息，收到文本/动作/字幕时间轴。
3. 开发者在固定会话中复盘历史消息。

## 5. 功能需求

### FR-01 健康检查
- 提供 `/api/ai-gf/health` 与 adapter `healthz`。

### FR-02 会话登录与信息
- 支持 `session/login` 与 `session/info`。
- 支持 token 绑定会话。

### FR-03 聊天路由
- 聊天请求默认走 OpenClaw 会话路由。
- 输出必须映射为 UE 结构化字段。

### FR-04 固定单会话模式
- 支持 `AI_GF_SESSION_MODE=fixed`。
- 默认写入同一会话 ID（当前默认 `ai-gf-fixed-session`）。

### FR-05 兼容分流模式
- 支持 token 分流模式（`SESSION_MODE=token`），用于并行多用户场景。

## 6. 非功能需求

- **可靠性**：请求失败可定位（日志 + 错误码）。
- **可维护性**：核心逻辑集中在 `ai-girlfriend`，避免多仓分裂。
- **可观测性**：会话 ID、请求链路可追踪。
- **性能**：聊天接口在可接受延迟内返回（与模型响应耗时相关）。

## 7. 约束与风险

1. OpenClaw 不同入口（gateway/channel 与 agent CLI）会话可见性存在差异。
2. 热重载可能导致代码切换感知不稳定，必要时需完整重启验证。
3. 客户端若存在旧 adapter 进程，会导致“看似改了但未生效”。

## 8. 验收标准

- [ ] UE 端 `continue-game/new-game` 均成功。
- [ ] `chat` 返回结构化字段完整。
- [ ] 人设回复符合预期（示例：Clawra）。
- [ ] 固定会话模式下历史可持续累积并可定位。

## 9. 需求边界总结

该方案优先解决“**可用、可查、可维护**”三件事：
- 可用：链路稳定
- 可查：固定会话可追踪
- 可维护：主线收敛到 ai-girlfriend
